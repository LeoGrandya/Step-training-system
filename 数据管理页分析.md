# 数据管理页现状分析

> 日期：2026-06-07  
> 分析范围：数据管理页 10 个模块与系统其他部分的连接关系

---

## 架构概述

数据管理页是一个**数据库直连管理后台**，绕过业务逻辑直接对 MySQL 的 10 张业务表做 CRUD。

核心机制是**数据驱动**：[adminData.js](frontend/src/services/adminData.js) 定义每个模块的 key、字段、列、外键 lookup，页面根据配置自动渲染表单和表格，不为每个模块单独写 UI。

左边 10 个模块对应 5 个人分工：

| # | 模块 | 负责人 | 表 |
|---|------|-------|-----|
| 1 | 受试者 | 陈彦竹 | `subjects` |
| 2 | 基础步伐 | 陈彦竹 | `footwork_types` |
| 3 | 自定义跑动序列 | 雷润华 | `route_definitions` + `route_steps` |
| 4 | 训练配置 | 金彦廷 | `training_configs` |
| 5 | 训练视频 | 金彦廷 | `training_videos` |
| 6 | 运动学数据 | 许婉琪 | `kinematics_datasets` + `kinematics_frame_metrics` |
| 7 | 效果评估 | 郝雨萱 | `evaluation_records` |
| 8 | 账号 | 金彦廷 | `accounts` |
| 9 | 角色 | 金彦廷 | `roles` |
| 10 | 权限 | 金彦廷 | `permissions` |

## 逐模块分析

### 1. 受试者（subjects）— 陈彦竹

- **管理页**：✅ 完整 CRUD，搜索，分页
- **系统关系**：全链路"人"的来源——训练配置选受试者、训练视频关联受试者、分析任务归属受试者、效果评估针对受试者、报告按受试者查询
- **⚠️ 问题**：之前 AuthView 登录注册走此表（`/api/users`），刚改成 `accounts` 表。但 `AnalysisView` 的 `ensureUser()` 仍在调 `/api/users` 找受试者。两套"人"未统一

### 2. 基础步伐（footwork_types）— 陈彦竹

- **管理页**：✅ 完整 CRUD，搜索，分页
- **系统关系**：被训练配置、自定义跑动序列、效果评估外键引用。种子数据 6 条：跨步、并步、撤步、两点、三点、四点
- **⚠️ 问题**：训练页"预置步伐"下拉来自 `presetSteps.js` 前端写死，管理页新增的步伐不会出现在训练页下拉

### 3. 自定义跑动序列（routes + route_steps）— 雷润华

- **管理页**：✅ 完整 CRUD（routes + steps），搜索，分页
- **系统关系**：与训练页"自定义库"下拉互通。管理页建路线 → 训练页可选；训练页保存 → 写入此表。每条路线可有多条步骤（宫格号、顺序、停留时长、节奏）
- **⚠️ 问题**：`actionRequirements` 和 `rhythm_json` 在训练页未完整消费；旧版 `/api/custom-footworks` 兼容接口仍在使用

### 4. 训练配置（training_configs）— 金彦廷

- **管理页**：✅ 完整 CRUD，外键校验，搜索，分页
- **系统关系**：分析页参数面板有训练配置下拉，选好后上传时带入 `trainingConfigId`。但 pipeline 本身不读配置——只是元数据标签
- **⚠️ 问题**：配置不参与实际计算；多维统计缺失

### 5. 训练视频（training_videos）— 金彦廷

- **管理页**：只读展示
- **系统关系**：分析上传时由 `v1.py` 自动写入左右视频路径和原始文件名
- **状况**：✅ 已对接上传流程

### 6. 运动学数据（kinematics_datasets + kinematics_frame_metrics）— 许婉琪

- **管理页**：只读展示
- **系统关系**：分析 pipeline 完成后自动写入产物索引（4 份 CSV 路径 + 摘要 JSON）
- **⚠️ 问题**：帧指标查询接口存在但落库逻辑需 pipeline 配合

### 7. 效果评估（evaluation_records）— 郝雨萱

- **管理页**：✅ 完整 CRUD，8 个组合过滤维度
- **系统关系**：关联受试者、分析任务、运动学数据、步伐、路线。pipeline 完成后可自动写入评估记录和报告
- **状况**：后端和报告链路基本完整

### 8. 账号 / 角色 / 权限（accounts / roles / permissions）— 金彦廷

- **管理页**：✅ 完整 CRUD
- **系统关系**：刚接入的 RBAC 系统。`accounts` 存 `account` + `username` + `password_hash`；`roles` 和 `permissions` 定义 API 访问控制
- **⚠️ 问题**：权限网关 `rbac_guard.py` 已写但默认关闭（`API_V1_RBAC_ENABLED` 未设），实际不生效
