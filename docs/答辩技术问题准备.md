### Q1. 为什么用 MediaPipe 做姿态检测，不用 OpenPose / MMPose / YOLO-Pose？

**完整答案：**

我们在项目启动阶段（2024 年初）对市面上主流的人体姿态检测方案进行了系统调研，最终选择 MediaPipe 是基于三个维度的综合考量。

首先是部署复杂度。我们这个系统的目标用户是运动实验室和体校，他们的电脑大多是普通的 Windows 台式机，没有独立 GPU，运维人员也不一定是计算机专业的。MediaPipe 只需要 `pip install mediapipe`，加上一个 `.task` 模型文件，就能在纯 CPU 环境下运行。OpenPose 需要编译 Caffe 或者 CMake 依赖，MMPose 依赖 PyTorch 加 MMCV 全家桶，这两个方案在 Windows 上的安装流程非常痛苦，而且对非专业用户来说几乎不可维护。我们项目设计了一个 `一键启动.bat` 脚本，就是希望用户双击就能开始用，这个需求直接把 OpenPose 和 MMPose 排除了。

其次是速度与精度的权衡。MediaPipe Pose Landmarker 的 Full 模型在 CPU 上可以达到 25-30 FPS，对 33 个身体 landmarks 的检测精度在 PCK@0.5 这个指标上大约在 85% 到 90% 之间。对于乒乓球步法分析来说，我们主要关心的是下肢大关节——髋、膝、踝——这些点在 MediaPipe 上的检测是比较稳定的。手指、面部这些精细 landmarks 我们并不需要。更重要的是，MediaPipe 输出的单目 3D 坐标我们是不用的——它在单目场景下估计的深度非常不靠谱，我们把它丢弃，真正的 3D 坐标通过双目相机标定加三角测量来获得。也就是说，MediaPipe 在我们系统里只负责 2D landmark 检测，而这是它最擅长的。

第三是架构上的可替换性。我们在 `pose3d_service.py` 里封装了一层 Service 抽象，未来的检测器可以替换而不影响下游 pipeline。如果现在重新选型，我会考虑 YOLOv8-pose 或者 RTMPose，因为它们都支持 ONNX 导出，可以在 CPU 上跑，而且精度比 MediaPipe 更高。选 MediaPipe 不是因为它最好，而是因为它在当时是最适合我们部署条件的方案。

### Q2. 为什么用 Flask 而不是 FastAPI？FastAPI 性能更好、文档自动生成，不是更"现代"吗？

**完整答案：**

FastAPI 确实在性能、类型安全和 API 文档方面优于 Flask，这一点我不否认。但我们选择 Flask 有四个具体原因。

第一，也是最关键的一点，FastAPI 的异步优势在我们的场景下基本用不上。FastAPI 的 `async/await` 模式解决的是高并发短连接的问题，比如聊天应用、实时推送，几千个用户同时发请求。但我们的系统核心不是高并发 API，而是长时间运行的 CPU 密集型分析任务。一个分析 job 跑 40 到 230 秒，全程占满 CPU。对这种任务来说，你用什么框架处理 HTTP 请求其实没区别，瓶颈在计算，不在 IO。我们用的是同步线程池来执行分析，Flask 的同步模型完全够用。

第二是团队因素。这个项目由 5 个人并行维护，按模块分工，Flask 的学习曲线比 FastAPI 低。FastAPI 的依赖注入、Pydantic 模型验证、Type Hints 强制要求这些特性，对习惯写 Python 脚本的同学来说是需要额外学习成本的。Flask 一个装饰器加一个函数就能定义路由，上手快，出活快。

第三是历史兼容性。系统里还有一套旧版的静态 HTML 页面，通过 iframe 加载，Flask 的 `send_from_directory` 和 Jinja2 模板对这一套兼容得很好。

第四，我们并没有把门关死。系统预留了 `backend/api_v1/` 的 Blueprint 架构，如果未来 API 面扩大，可以在 Blueprint 内部逐步迁移到 FastAPI，或者用 Quart——它是 Flask 的异步等价实现，API 完全兼容。这是一个务实的渐进演进策略，而不是"选 Flask 就永远只能用 Flask"。

补充一点，Flask 开发服务器确实是单线程的，但我们生产部署用的是 `waitress` 或者 `gunicorn`，可以开多 worker。而且分析 pipeline 走的是独立的 `AnalysisExecutor` 线程池，不受 Flask 请求线程的影响。

### Q3. 三角测量用的是 `cv2.triangulatePoints`，这是一个基础的 DLT 方法。为什么不用 Bundle Adjustment 或者深度学习的方法？

**完整答案：**

这个问题触及了计算机视觉里一个很重要的原则——方法的复杂度应该和问题的难度匹配。我们用的是 pre-calibrated stereo cameras，也就是说两个相机的投影矩阵 P1 和 P2 是已知且固定的。在这种条件下，`cv2.triangulatePoints` 的 DLT，也就是直接线性变换方法，其实是数学上的最优解。

为什么？因为三角测量本质上是一个线性方程组求解问题。已知左图像点 (u₁, v₁)、右图像点 (u₂, v₂) 和两个 3×4 的投影矩阵，求 3D 点 X。DLT 把这个写成 AX=0 的形式，然后对 A 做 SVD 分解，取最小奇异值对应的奇异向量作为解。这个过程没有迭代、没有初值依赖、不会收敛到局部最优——它一步到位。在投影矩阵准确的前提下，DLT 的数值误差主要来自 landmark 检测误差，而不是三角测量算法本身。

那为什么不加 Bundle Adjustment？BA 同时优化 3D 点和相机参数，但我们的相机参数已经通过预标定固定了，每帧的 3D 点是独立计算的。对单帧的三角测量做 BA，本质上等价于在已经确定的最优解附近微调，收益极低。BA 的价值在于"多帧联合优化"，比如 SLAM 里几百帧共享同一组相机参数和 3D 地图点。我们的场景是一帧一帧独立处理，没有这个联合优化的必要。

深度学习的方法，比如 TriViT 或 PoseFormer，直接从 2D landmarks 回归 3D 姿态。这类方法的问题是它们依赖大规模训练数据，而乒乓球步法是一个长尾场景——低姿态、快速变向、自遮挡多，不在任何公开数据集的分布内。你拿 COCO 或 Human3.6M 上训练的模型来跑乒乓球，它会按"走路"或"站立"的先验去猜测被遮挡的腿，猜出来的结果可能看起来很平滑但不真实。几何方法没有这个先验偏见，遮挡了就标记 NaN，不会被模型"脑补"。

当然，几何方法也有局限。2D landmark 的检测误差会被直接传播到 3D。我们的改进方向不是换三角测量方法，而是提高 2D landmark 精度——换更好的检测器——以及加时间域的平滑滤波。

### Q4. 双机位同步用的是音频互相关，为什么不用视觉特征匹配？比如 SIFT 加 RANSAC？

**完整答案：**

这两种方法解决的是同一个问题的不同版本。视觉特征匹配的前提是两个相机能看到同一组特征点，也就是说它们的视场必须有足够大的重叠区域。在乒乓球训练的拍摄场景中，左右两个机位的角度差接近 90 度——左相机几乎正对运动员的右侧面，右相机正对左侧面。这种情况下，两个画面里的共同特征非常少，SIFT 能匹配到的点可能只有几个，而且都集中在背景的边缘区域，匹配质量很差。

音频同步则完全没有这个要求。声音是全局信号，不管你相机朝哪个方向，环境里的蜂鸣器声音、击球声、脚步声都会被两个相机录进去。我们只需要各提取一段单声道 8kHz 的音频，然后用 `scipy.signal.correlate` 做 FFT 互相关，复杂度是 O(n log n)。对一段 45 秒的音频，这个计算耗时不到 0.1 秒。

而且我们的场景有一个天然的同步标记——乒乓球训练使用 3 秒蜂鸣器信号。系统里专门实现了蜂鸣器检测函数，能识别出持续 2.5 秒以上、振幅超过最大值 40%、能量密度超过 70% 的音频区间，然后取这个信号结束的那个采样点作为对齐基准，对两个视频做裁剪。这是领域知识指导算法设计的典型案例——通用的音频互相关可以处理任意声音，但加上蜂鸣器先验可以把对齐精度提高一个数量级。

当然，音频同步也有局限。如果两个相机距离非常远，比如超过 50 米，声速延迟大约是每米 3 毫秒，会引入 1 到 2 帧的误差。如果环境噪杂到听不到蜂鸣器声，信号会被淹没。这两种情况下系统会 fallback 到手动偏移量，通过 `sync_overrides` 参数传入。我们设计的时候已经把降级路径考虑进去了。

### Q5. 为什么用 ECharts 做图表，不用 D3.js 或 Chart.js？

**完整答案：**

前端图表库的选择主要看三个维度：图表类型的覆盖度、大数据量的渲染性能、开发效率。

第一是图表类型。我们的报告页面有 12 个图表组件，包括综合对比图、位移轨迹图、效率评估图、腾空参数图、足底压力图、步法热力图、关节生物力学图、周期时序图、雷达指标图、速度加速度图、统计概览图、台面站位图。涉及的图表类型有折线图、雷达图、热力图、散点图、仪表盘、柱状图等。ECharts 5.6 对这些类型全部内置支持，拿来就能用。如果用 D3.js，每种图表类型都要从头写渲染逻辑，12 个组件 12 套代码，开发量是 ECharts 的 3 到 5 倍。

第二是数据量。单次分析产生大约 900 帧的运动学数据，总共 12 类指标。这些数据要能在前端缩放、刷选、hover 查看细节。ECharts 内置的 `dataZoom` 组件和 `tooltip` 系统在这方面非常成熟，处理千级数据点完全没有压力。Chart.js 在数据量超过几百个点时，Canvas 渲染性能开始下降，而且交互组件需要额外插件。

第三是开发生态。ECharts 的文档、中文社区、示例都非常丰富，团队里 5 个人有 3 个人之前用过。在大屏展示场景下，ECharts 的渲染效果一直很好。而且它和 Vue 3 的集成非常自然——我们在 `onMounted` 里 `echarts.init`，在 `watch` 里 `setOption`，在 `onUnmounted` 里 `dispose`，生命周期非常清晰。

不用 D3.js 的另一个原因是，D3 的优势在于制作"没有现成库能画的定制可视化"。但我们的需求是标准的运动科学图表——散点图、热力图、轨迹图——这些都是"有现成库能画"的类型。花时间在 D3 上重新发明轮子不值得。

### Q6. 为什么用 Butterworth 滤波器而不用卡尔曼滤波？卡尔曼滤波不是更"高级"吗？

**完整答案：**

这是一个很多人都会问的问题，因为卡尔曼滤波在追踪和导航领域确实非常流行。但"高级"不等于"适合"。

核心区别在于，Butterworth 是一个离线、零相位的滤波器，而卡尔曼滤波是一个在线、因果的滤波器。所谓离线，就是我们拿到全部帧的数据之后一次性处理，所以可以用 `scipy.signal.filtfilt`——先把信号正着滤一遍，再倒过来滤一遍。这样做的结果是输出信号的相位和原始信号完全一致，不会有时延。卡尔曼滤波只能看到当前帧和过去的帧，输出的估计天然会有一个滞后。对于离线分析来说，这个滞后是一个纯粹的性能损失，没有任何好处。

第二个问题是运动模型。卡尔曼滤波的核心优势是融合物理模型和观测数据。比如追踪一架飞机，你可以写一个很精确的匀加速运动模型作为预测，然后用雷达数据来修正。但乒乓球步法运动——急停、变向、交叉步、跳跃——很难用一个简单的线性状态转移方程来描述。你如果强行套一个匀速模型，当运动员突然变向时，卡尔曼滤波会"不信"观测数据，坚持往之前的速度方向延伸，造成更大的误差。错误的运动模型比不用模型更糟糕。

第三个是调参成本。Butterworth 滤波器只有两个参数：截止频率和阶数。截止频率我们设在 6 到 8 Hz，这个基于运动科学的公认结论——人体自愿运动产生的最高频率大约在 5 到 8 Hz，高于这个频段的信号大概率是检测噪声而不是真实运动。阶数用 4 阶，在阻带衰减和计算稳定性之间取平衡。两个参数就能覆盖所有运动员和所有步法类型。卡尔曼滤波需要调过程噪声协方差矩阵 Q 和观测噪声协方差矩阵 R，没有运动捕捉 ground truth 作为参考的话，这些参数基本靠猜。猜出来的卡尔曼滤波不一定比调好的 Butterworth 好。

不过我必须承认，Butterworth 在两种场景下不够理想。一个是长遮挡，超过 5 帧的关键点缺失，线性插值会产生非自然的轨迹，Butterworth 会把这些插值后的轨迹当真实数据去滤波。另一个是冲击类运动——比如落地瞬间脚踝接触地面的高频加速度——会被 Butterworth 直接抹平。这两个问题是我们的后续改进方向，可以考虑自适应截止频率或者在遮挡区间引入运动学约束。

### Q7. 运动学参数的计算——关节力矩、做功、膝关节锁定指数这些——有什么简化假设？精度怎么保证？

**完整答案：**

这个问题很重要，因为我们报告里展示的力矩和做功值不能理解为"真实的牛顿米"和"真实的焦耳"。我来解释简化了什么、为什么可以这样做、以及精度大概在什么范围。

我们用的是简化逆动力学方法，不是完整的刚体动力学仿真。具体简化包括以下几层。

第一层，身体被简化为 6 个独立链段，两条腿各 3 段——大腿、小腿、足部。每个链段被当作点质量处理，质量集中在质心位置。质心位置、链段长度、链段质量全部来自 Dempster 人体测量模型，这是一个基于身高和体重的经验公式。比如大腿质心大约在髋关节往下 43% 大腿长度的地方，小腿质心在膝关节往下 43% 小腿长度的地方。这些百分比是统计平均值，对任何一个具体的运动员来说有 ±5% 到 ±10% 的个体偏差。

第二层，力矩的计算公式是 τ = I × α，即转动惯量乘以角加速度。我们忽略了重力矩，也就是链段自身重量产生的力矩；忽略了关节相互作用力矩，比如髋关节的运动会影响膝关节的受力；忽略了地面反力，因为我们的系统只测了运动学，没有测力台。这意味着我们算出来的力矩只捕捉了使关节加速的那部分力矩，而不是关节真实受到的合力矩。真实力矩应该更大。

第三层，转动惯量 I 的估算用的是 I = m × d² 的点质量近似。大腿质量取体重的 10%，质心到关节距离取大腿长度的 45%。这些参数来自人体测量学的平均值，对大多数成年人来说误差在可接受范围内。

那么精度到底多少？我们没有和金标准做对比验证，所以无法给出一个准确的数字。但可以做一个合理的推断。运动学参数——角度、速度、位移、加速度——直接来自 3D 坐标的数值差分算出来的，精度取决于三角测量精度。在我们的拍摄条件下，3D 坐标的误差大约在 ±1 到 ±2 厘米。转换成角度，对一个 40 厘米长的大腿段来说，1 厘米的端点误差大约对应 1.4 度的角度误差。所以运动学参数的相对精度还是比较高的。

动力学参数——力矩和做功——的误差要大得多，主要来自模型简化假设。我个人的估计是，绝对力矩值的误差可能在 30% 到 50% 这个量级。但这不意味着这些数字没有用。因为系统的核心场景不是给出一个绝对准确的力矩数值，而是做"同一个运动员前后对比"或者"不同步法之间的横向比较"。在相对比较中，模型偏差对所有数据的影响是一样的，所以相对排名是可靠的。

我会建议在报告界面上给动力学参数加一个标注，写明"估算值，供参考"，提示用户不应将这些数字与临床测力台数据直接对比。

### Q8. 步法分段的有限状态机是怎么设计的？为什么不用 LSTM 之类的机器学习方法来分类？

**完整答案：**

我们的状态机设计参考了乒乓球步法的标准教学模型。一个完整的步法周期被分解为三个阶段。

第一个阶段是准备移动，运动员看到球的方向，身体开始启动。判断依据是 COM（质心）水平速度超过启动阈值，默认是 0.5 m/s。第二个阶段是移动到位，运动员移动到目标位置后减速停下。判断依据是 COM 速度降到静态阈值以下，同时 9 宫格位置与目标格子匹配。第三个阶段是恢复归位，运动员打完球后回到预备位置。判断依据是 COM 回到初始位置的一定范围内。

状态之间的转移逻辑是确定性的——在每个状态内，系统持续检查是否满足切换到下一状态的条件。所有的阈值都是可配置的，存储在 `user_params.py` 里，一共 9 个参数。

那为什么不用机器学习呢？主要是三个原因。

第一是标注成本。一个有监督的 LSTM 或 Transformer 分类器需要逐帧标注步法阶段。一段 20 秒的训练视频大约 1000 帧，每一帧都要标上是在"准备"、"移动"还是"恢复"。如果要覆盖不同运动员、不同步法类型，至少需要几百段视频的标注。在乒乓球这个细分领域，没有公开的标注数据集，全靠自己标。这成本太高了。

第二是可解释性。状态机出错了，我可以定位到具体的阈值。比如说"速度阈值设太高了，导致慢速启动没有被检测到"，改一个数字就行。机器学习模型出错了，我需要去分析注意力权重或者用 SHAP 值来解释，而且解释了也不一定能修。对于教练和运动员来说，他们需要理解"为什么这个步法被判定为有延迟"，这个解释如果是基于物理量（速度、位置、时间）的就很容易理解，如果是基于"神经网络的第 5 层第 32 个神经元的激活值"就完全没法解释。

第三是跨运动员的泛化。状态机用物理量做判据——速度和位置——这些不受个人风格的影响。不管你是高个子还是矮个子，你的 COM 速度降到 0.3 m/s 以下就说明你基本停住了。但机器学习模型如果用 5 个运动员的数据训练，它在第 6 个运动员身上可能表现很差，因为它学到的是"这个运动员停下来的姿势长什么样"，而不是"停下来的物理条件是什么"。

有一个诚实的局限要承认：当前状态机对非标准的混合步法——比如交叉步加并步的组合——识别效果不好。未来的方向可以考虑"状态机做候选段生成，机器学习做精细分类"的混合方案，各取所长。

### Q9. 双目标定的精度怎么保证？标定误差对最终的 3D 坐标影响有多大？

**完整答案：**

我们的标定数据来自预标定流程，用 MATLAB Stereo Camera Calibrator 或者 OpenCV 的 `stereoCalibrate` 函数完成。标定输出是左右相机的内参矩阵、畸变系数、以及两个相机之间的旋转矩阵 R 和平移向量 T，最终封装为两个 3×4 的投影矩阵 P1 和 P2，存储在 `stereo_params.json` 里。系统在运行时不重做标定，只读取这个 JSON 文件。

标定精度可以通过重投影误差来评估。我们的 pipeline 在 pose3d 阶段会计算重投影误差，通常在 0.3 到 0.8 像素的范围内。这个数字意味着，把三角测量得到的 3D 坐标重新投影回左右两个相机画面，与 MediaPipe 检测到的原始 2D 坐标之间的差距不到 1 个像素。这说明三角测量在数值上是自洽的。

但自洽不等于绝对准确。标定误差传播到 3D 坐标大约是这样一个关系：在我们的拍摄条件（3 到 5 米拍摄距离，基线约 1 米）下，2D 图像上 1 个像素的 landmark 检测误差，对应 3D 空间中大约 1 到 2 厘米的误差。如果标定本身不准——比如标定板没有被正确检测、或者标定时相机位置和实际拍摄时不一致——那么重投影误差会显著增大，3D 坐标中会引入系统性的偏置。

系统提供了一些补偿手段。一是支持手动覆盖标定文件，通过上传 `stereo_params.matlab.json` 来替代默认参数。二是地面对齐，利用"脚接触地面"的先验知识，把所有 3D 点的 Z 坐标整体偏移，使得地面接触点的 Z 坐标均值为零。三是坐标轴符号翻转，如果标定输出的坐标轴方向和实验室的实际坐标不一致，可以配置翻转。

如果要保证标定的长期稳定性，我建议至少每月检查一次。检查方法很简单——拍一段已知尺寸的标定板或者参照物视频，跑一次分析，看系统输出的距离和实际距离的偏差是否在可接受范围内。如果偏差开始增大，就说明需要重新标定了。

## 三、系统架构

---

### Q10. 为什么分析任务用线程池而不是 Celery 加 Redis 这种任务队列？

**完整答案：**

Celery 加 Redis 解决的核心问题是"如何在多台机器之间分发和调度任务"。我们的系统目前的目标部署是单台 Windows 台式机，也就是说所有分析任务都在同一台机器上执行。在这种情况下，引入 Celery 和 Redis 不会带来任何好处，却会增加两个额外的服务依赖。

具体来说，Celery 需要你启动一个 Redis 服务或者 RabbitMQ 服务作为消息中间件，需要启动 Celery Worker 进程来消费任务，需要管理进程的启动、监控和崩溃重启。在我们的场景里——`ANALYSIS_MAX_WORKERS` 默认等于 1，因为单个 MediaPipe 实例已经占满一个 CPU 核心——并不存在"多台机器协同"的需求。用 ThreadPoolExecutor 管理一个线程池，配合一个 Semaphore 控制排队数量，代码只有 50 行，逻辑一目了然，不需要引入任何外部依赖。

而且我们的 `JobStore` 已经在内存和磁盘之间做了双写，`meta.json` 持久化了任务状态。如果进程崩溃重启，`_recover_stale_if_needed` 方法会把磁盘上的 queued 状态的旧任务标记为 INTERRUPTED。这个机制虽然不如 Celery 的任务重试和 ETA 调度那么完善，但对单机场景来说已经足够。

我还想强调一个设计原则：架构的复杂度应该和目标规模匹配。一个在 5 台机器上部署的系统需要消息队列，一个在 1 台机器上部署的系统不需要。过早引入分布式架构组件是一种过度工程。我们设计的 `AnalysisExecutor` 是对标准库 `ThreadPoolExecutor` 的薄封装，如果未来确实需要水平扩展，只需要把 `submit()` 方法的实现替换为 Celery 的 `task.delay()`，调用方从 `start_pipeline_thread` 到所有上层路由完全不用改。API 是稳定的。

如果评委继续追问"那如果多个教练同时使用呢"，我会回答：当前 `ANALYSIS_MAX_WORKERS` 可以配置到 8，配合 `ANALYSIS_MAX_QUEUE` 配置到 8，总共可以处理 16 个并发或排队任务。对于一个实验室里 1 到 2 个教练同时使用的场景来说，这个容量足够。如果未来要扩展到几十人规模的体校场景，我会考虑引入任务队列，可能是 Celery，也可能是更轻量的 RQ。

### Q11. 为什么做模块缓存系统？缓存键怎么设计的？缓存什么时候过期？

**完整答案：**

缓存是我们在性能优化上最成功的一个设计决策。从系统运行数据来看，首次无缓存时，sync 阶段需要 3 到 10 秒，pose3d 阶段需要 120 到 240 秒。但同一个视频你可能会分析多次——比如说你第一次用"快速"档位跑了一遍，觉得结果不错，想换"质量"档位再跑一遍看细节；或者你发现了代码里一个 bug，修完之后需要重新跑一遍。没有缓存的话，每次都要等好几分钟。有了缓存，sync 和 pose3d 阶段从几百秒降到 1 秒左右，直接复用之前计算好的对齐视频和 3D CSV 结果。

缓存键的设计遵循了内容寻址原则，不是基于文件名，而是基于输入内容的哈希。sync 阶段的缓存键包含：左右视频文件的 MD5 校验和、文件角色标签（标明哪一个是左相机哪一个是右相机）、以及所有可配置参数——video_mode、crf、max_audio_seconds。pose3d 阶段的缓存键额外包含了 max_frames、frame_stride、后处理参数等。这意味着只要输入内容和参数不变，缓存一定命中；任何一个变了，缓存自动失效。你永远不用担心"用错了缓存"这个问题。

关于缓存过期策略——我们目前没有。这不是设计缺陷，是故意的。缓存在 `artifacts/immutable/` 目录下，"immutable"的意思就是说这些文件一旦写进去就不再修改。不存在"缓存需要刷新"的场景，因为如果输入或参数变了，缓存键就变了，会产生一个新的缓存条目，旧的不受影响。用户如果觉得磁盘空间不够了，手动删除整个目录就行。

但这不是说完美的。缓存系统目前没有做容量上限控制，如果用户上传了很多不同的视频，缓存目录会持续增长。未来可以加一个 LRU 淘汰策略或者容量上限。但对于当前的使用规模——一个实验室一天处理不超过 10 段视频——暂时不需要。

### Q12. 前端为什么没有用 Pinia 或 Vuex 管理状态，而是直接用 localStorage？

**完整答案：**

Pinia 和 Vuex 解决的是复杂 SPA 中"多个组件之间共享响应式状态"的问题。但在我们系统里，需要跨组件共享的状态其实非常少。

具体来说，前端需要持久化的状态只有这几个：当前选中的受试者 ID、最近点击过的训练配置、UI 主题偏好、引导教程是否已完成。全部加起来不超过 8 个键。这些状态的消费者通常只有一两个组件，不存在"A 组件改了一个值，B、C、D 组件都需要同步响应"这种复杂场景。

我们用了 Vue 3 的 Composable 模式来实现轻量级的状态管理。比如说 `useAnalysisJob` 这个 composable，它内部封装了 localStorage 的读写逻辑，对外暴露响应式的 ref。从使用者的角度看，它就是 `const { jobId, status } = useAnalysisJob()`，跟用 Pinia Store 的体验差不多。区别在于没有引入额外的库，也没有定义 Store、Action、Getter 这些概念。

我们给所有 localStorage 键加了 `ai_sport_lab.*` 前缀，这是一个好的工程习惯，防止同域名下不同应用的键冲突。

如果项目继续增长，组件之间的状态交叉依赖变得复杂，我会考虑引入 Pinia。Pinia 的 `persist` 插件可以直接对接 localStorage，迁移成本很低。但现在这个阶段，引入 Pinia 的收益配不上引入一个额外依赖的成本。

### Q13. 数据库为什么默认用 SQLite，但同时又保留了 MySQL 的支持？

**完整答案：**

这是一个"按部署场景选择存储"的设计。SQLite 和 MySQL 各自服务不同的部署模式。

SQLite 是为本地单机部署设计的。它的好处非常直接：零配置。启动脚本里不需要"先装 MySQL、再建库、再跑 migration"，Python 自带了 sqlite3 模块，Flask 启动时自动在 `web_1/data/` 下创建 `app.db` 文件。用户双击 `一键启动.bat` 就能用。备份也简单，复制一个文件就行。对于实验室的场景——一个教练、一台电脑、一个数据库文件——SQLite 完全够用，WAL 模式加持下读写并发也不差。

但 SQLite 有一个根本性的瓶颈：写操作是全局串行化的。任何时候只有一个线程能写。在我们的系统里，分析线程会频繁更新任务状态——从 queued 到 running，从 sync 到 pose3d 到 kinematics——每秒可能触发多次写操作。而 Flask 的 Web 请求线程也会并发读任务状态。在 WAL 模式下，读操作不会被写操作阻塞，但多个同时的写操作会排队。在单 worker 场景下这不是问题，但如果以后 `ANALYSIS_MAX_WORKERS` 调到 4 或者 8，写冲突就会显现出来。

MySQL 解决了这个并发写入的瓶颈，因为它支持行级锁和真正的并行写入。MySQL 也允许多个应用实例共享同一份数据——如果你在前面加一个负载均衡器，多个 Flask 实例可以指向同一个 MySQL 服务器。这就是服务器部署场景下的标准做法。

切换方式是环境变量。你设置 `DATABASE_URL=mysql+pymysql://user:pass@host/db`，系统自动走 MySQL + PyMySQL 驱动；不设置的话默认走 SQLite。Alembic 迁移脚本只对 MySQL 生效，SQLite 走 `db.create_all()` 自动建表。这个双轨设计让我们在"本地开发方便"和"服务器部署可靠"之间不需要做取舍。

### Q14. 系统的精度怎么验证的？有没有和金标准对比过？

**完整答案：**

坦白说，和金标准的交叉验证是我们目前最大的一个缺失。所谓金标准，指的是 Vicon 或 Qualisys 这类光学运动捕捉系统，它们用十几个红外摄像头追踪反光标记球，3D 位置精度可以达到亚毫米级。理想的做法是，同一个运动员同时被我们的双相机系统和 Vicon 系统记录，然后对比我们输出的关节角度、COM 速度与 Vicon 输出的差异，计算 RMS 误差和 Bland-Altman 一致性。

由于条件限制，我们还没有做这一步。但我可以说明我们现有的间接验证手段。

第一是重投影误差。三角测量结束后，我们把算出来的 3D 坐标重新投影回左右两个相机画面，看和 MediaPipe 原始检测的 2D 坐标差多少。这个误差通常落在 0.3 到 0.8 像素之间，在 3 到 5 米拍摄距离下对应大约 1 厘米以下的 3D 误差。这说明三角测量在数值上是自洽的，相机模型的参数是合理的。

第二是骨骼长度一致性。人体的股骨和胫骨在运动过程中长度是不变的——这是一个很强的几何约束。我们可以在系统内部做一次校验：对每一帧分别计算骨骼长度，如果各帧之间有大幅波动，那就说明三角测量在这一帧出了问题。虽然我们目前没有把这项校验集成到 pipeline 里，但它是一个很容易实现的精度验证手段。

第三是物理合理性检查。比如一个乒乓球运动员的水平移动速度峰值大约在 3 到 5 m/s，如果我们系统输出 10 m/s 的 COM 速度，说明有问题。这种基于常识的 sanity check 是每一段视频分析后都可以做的。

第四是和其他研究的间接对照。运动科学文献里有大量关于乒乓球步法的运动学数据。我们可以把系统输出的参数范围和文献里的数值做对比——如果数量级一致，说明没有系统性偏差。

我强烈建议下一步把这个金标准验证作为最高优先级任务来做。具体方案是与体育院校的运动生物力学实验室合作，在他们的 Vicon 实验室里录制标定好的同步数据，产出一份精度报告。这份报告的关键指标包括：关节角度的 RMS 误差、COM 速度的一致性系数、以及步法分段的时间误差。

### Q15. 怎么处理遮挡问题？比如运动员转身时一条腿被另一条腿挡住的情况。

**完整答案：**

遮挡是纯视觉方法无法回避的问题。当运动员双腿交叉或者转身时，远离相机的那条腿在某些视角下会完全被遮挡，MediaPipe 检测不到 landmarks，输出 NaN 或者极低置信度的坐标。我们的处理策略是多级降级的。

第一级是 MediaPipe 自身的可见性筛选。每个 landmark 都带有一个 presence 分数，我们设了阈值 0.5，低于这个值的坐标直接丢弃。

第二级是 MAD 异常值检测，MAD 就是中位数绝对偏差。在滑动窗口内，如果某个帧的 landmark 位置偏离窗口中位数超过 3.5 倍 MAD，它会被判定为异常值并替换为 NaN。这一步不是针对遮挡的，但可以顺带处理那些"MediaPipe 给出了一个坐标但坐标是错的"的情况。

第三级是线性插值。对于连续的 NaN 段，如果段长度不超过可配置的最长间隔帧数，就用两端的有效值做线性插值补全。对于短遮挡——两到三帧，大约 40 到 60 毫秒——线性插值的效果是可以接受的，因为这段时间内关节的移动量很小。

第四级是 Butterworth 分段滤波。我们不是对整个信号做滤波，而是先找到有效段，对每一段分别滤波，然后再拼回去。这样做可以避免 NaN 值污染相邻的有效数据。

但必须承认，对于连续超过 10 帧的严重遮挡，线性插值会产生非物理的轨迹——关节不可能沿着一条直线穿过遮挡区。这是当前方法最明显的局限。

如果要改进，有两条路可以考虑。一是人体运动学约束，利用"股骨长度恒定"、"膝关节的屈伸角度范围有限"这些先验，把遮挡区间的坐标估计变成一个带约束的优化问题。二是多假设跟踪，用粒子滤波同时维持对被遮挡关节的多个可能位置的假设，等重新可见时再做后验修剪。但这些方法都是双刃剑——引入了更多的计算复杂度，需要更仔细的调参。

### Q16. 同一段视频跑两次分析，结果一致吗？

**完整答案：**

一致。整个 pipeline 中所有算法都是确定性的。

MediaPipe 在 CPU 上推理是确定性的——同样的模型文件、同样的输入图像、同样的推理配置，输出完全相同。三角测量是纯线性代数运算，没有随机性。Butterworth 滤波是数学公式。分段状态机是确定性的规则判断。所以给定相同的输入，两次运行的输出一定逐位相同。

一个可能引入差异的地方是同步阶段的视频重编码。如果 sync_video_mode 设为 `reencode`，ffmpeg 会用 libx264 重新压缩视频。虽然 x264 在相同参数下是确定性的，但如果不同机器上安装的 ffmpeg 版本不同，理论上可能产生微小的像素级差异。在 `copy` 模式下，视频数据完全不做重新编码，只裁剪时间轴，这是 100% 可重复的。

缓存系统的表现本身也验证了可重复性。ArtifactCache 是基于内容哈希工作的——如果两次输入完全一致，缓存命中，直接返回缓存结果。如果两次运行产生了不同的结果，缓存不会命中，但这种情况在我们的测试中从未发生过。

### Q17. 相机标定之后能管多久？什么情况下需要重新标定？

**完整答案：**

相机标定的有效期取决于相机的物理稳定性。如果相机被牢牢固定在三脚架上、焦距环用胶带锁住、没有人碰它，标定参数可以稳定几周甚至几个月。但只要发生以下任何一种情况，就需要重新标定。

第一，相机位置被移动了。哪怕只是轻轻碰了一下三脚架，厘米级的位移就足以在 3 到 5 米拍摄距离上引入可观测的 3D 误差。

第二，焦距或光圈被改变了。变焦镜头如果被人拧了缩放环，内参矩阵中的焦距会变化，标定直接失效。

第三，环境温度剧烈变化。热膨胀会导致三脚架或相机支架微动。虽然这个效应在室内通常可以忽略，但如果实验室是朝西的、夏天下午阳光直射到设备上，温度变化确实可能影响标定。

我们建议的正常维护周期是一个月检查一次。检查方法很简单——拍一段标定板或者已知距离的参照物视频，跑一次分析，测量系统输出的距离和实际距离之间的偏差。如果偏差开始变大，就重新标定。

长期来看，如果项目要走向产品化，可以在系统中集成一个简易的在线标定验证功能。例如在画面中放置一个已知尺寸的固定标记物，系统每次分析时自动测量标记物的尺寸，如果偏差超过阈值就弹出"建议重新标定"的提醒。

### Q18. 5 个人并行开发，怎么管代码质量和协作？

### Q19. 怎么防止用户上传恶意文件或者超大视频导致系统崩溃？

**完整答案：**

坦白说，这是我们防御比较薄弱的环节，目前的防护主要依赖前端。

前端用了 `accept="video/*"` 限制文件选择器的可选类型，但这只是一个 UI 层面的引导——任何人都可以通过 curl 或者 Postman 绕过它，上传任意类型的文件到后端。后端目前的代码没有做文件魔术字校验，也就是说你上传一个伪装成视频的 exe 文件，后端也会照单全收并尝试调用 ffmpeg 去处理它。这是一个安全漏洞。

文件大小方面，Flask 默认没有设置 `MAX_CONTENT_LENGTH`，也就是说用户可以上传任意大的文件，可能撑爆服务器磁盘。分析 pipeline 也没有对视频帧数或时长设置上限——你上传一个 2 小时的视频，系统会尝试逐帧做 MediaPipe 检测，可能需要几个小时，而且会一直占用唯一的 worker 线程，导致其他用户的任务排队。

我们的防御机制有：`ANALYSIS_MAX_QUEUE` 环境变量限制排队任务数为 8，间接限制了同时提交的数量。视频处理中的 ffmpeg 子进程调用在 `video_sync` 模块里，如果有异常可以通过顶层 try/except 捕获。

如果要加固，我建议做这几个改进。第一，在 Flask 配置里设置 `MAX_CONTENT_LENGTH`，比如 500MB。第二，对上传文件的二进制流校验文件头魔术字，确认是真实的视频格式。第三，在分析 pipeline 入口处检查视频时长，超过可配置最大时长就直接拒绝。第四，对 ffmpeg 子进程设置 timeout，防止进程卡死。

这些改进都不复杂，主要是我们之前没有把安全作为优先级来考虑。现在如果要把项目交付给真实用户使用，这些是一定要补上的。

### Q20. 分析到一半服务器断电或者进程崩溃了，任务状态怎么恢复？

**完整答案：**

我们的恢复策略依赖于 `meta.json` 文件的磁盘持久化。Pipeline 在执行过程中，每次阶段切换时都会调用 `JOB_STORE.write_meta_file()` 把最新的状态写入磁盘。具体来说，每个 job 在创建时写入一次（状态为 queued），进入 sync 阶段时写入一次（状态为 running），进入 pose3d 阶段时再写入一次，以此类推。所以一个 job 的 meta.json 在整个生命周期中大约被更新 5 次。

当服务重启时，`JobStore.get()` 方法在内存中找不到 job 记录后，会尝试从磁盘的 `meta.json` 加载。加载时 `_recover_stale_if_needed` 方法会检查状态，如果状态是 queued 或 running，说明这个任务在进程中断时还没有完成，系统会将其标记为 failed，错误代码设为 INTERRUPTED，错误消息提示用户"分析中断：服务已重启或进程结束，请重新点击开始分析"。

这样做有几个好处。第一，前端不会永远展示一个"幽灵任务"，用户看到的是明确的失败状态和重试提示。第二，中间产物——已经计算好的同步视频和 3D CSV——保留在磁盘上。用户重新发起分析时，ArtifactCache 会命中这些已有产物，跳过重复计算。

但也有一些未覆盖的场景。线程池内部队列中的任务——已经通过 semaphore 检查但还没有被 worker 线程实际开始执行的——仅存在于内存中，进程退出后就永远丢失了。如果 meta.json 还没来得及写入就被断电，新的 job 目录虽然在磁盘上，但没有 meta.json 文件，重启后查询会返回 404。

`_write_meta_file_record` 方法用了 `tempfile.mkstemp` 加 `os.replace` 的方式做原子写入，这一点做得是对的——即使写入过程中断电，要么旧文件完整保留，要么新文件完整写入，不会出现写一半的损坏文件。

如果要进一步改进，我考虑加入阶段内的定期 checkpoint 机制，比如运动学分析每处理 100 帧就保存一次中间进度，这样即使断电也只丢失最后 100 帧的计算结果而不是全部从头开始。

### Q21. 这个系统最大的技术局限性是什么？

**完整答案：**

如果只选一个最大的局限，那就是我们没有和金标准运动捕捉系统做过精度验证。这导致报告中展示的所有运动学参数的绝对精度是未知的。

在这个大局限之下，我认为有三个维度上的具体局限。

精度维度上，除了缺乏金标准对标外，动力学参数——关节力矩、做功、关节能量效率——依赖的是简化的人体模型，理论误差可能在 30% 到 50% 量级。这些数字对临床诊断没有参考价值，但在同一个运动员的纵向对比中仍有相对意义。

鲁棒性维度上，严重遮挡场景是我们目前最薄弱的环节。当运动员转身或者双腿交叉超过 5 帧时，我们的线性插值策略会产生非物理的轨迹。而且 MediaPipe 对极端姿态（如深蹲、弓步）的检测精度会下降，这会影响下肢关节角度的准确性。

泛化性维度上，整个系统是为乒乓球步法分析高度特化的。步法分段的状态机逻辑——9 宫格站位模型、蜂鸣器检测、三段式步法模型——全部基于乒乓球的领域知识。如果要迁移到篮球、网球或田径项目，需要重写整个分段逻辑。这并不是一个"通用运动分析系统"，而是一个"乒乓球步法分析系统"。

### Q22. 如果给你 3 个月继续改进，你会优先做什么？

**完整答案：**

我会按这个优先级排。

第一个月，做金标准精度验证。与体育院校的运动生物力学实验室合作，在 Vicon 实验室录制双目视频和 Vicon 光学捕捉的同步数据。产出关节角度 RMS 误差、COM 速度一致性系数、步法分段的时间误差。这份精度报告是系统从"看起来能用"到"真的能用"的门槛。没有它，任何关于精度的说法都是猜测。

第三个月，加入近实时分析模式。当前系统是完全离线批处理的——上传整段视频，等几分钟，看结果。但教练更需要的是训练现场的即时反馈。我会把 pipeline 改成"边录边分析"，录够一定帧数就开始处理，输出简化指标——COM 移动速度、步法频率、双支撑时间比例——实时显示在训练现场的显示器上。不需要等完整视频录完就能看到初步数据。这个功能是教练在需求访谈中反复提到的。

### Q23. 为什么要搞缓存系统而不是用 GPU 加速？GPU 不是比缓存更快吗？

**完整答案：**

这是两个不同维度上的优化，解决的是不同的问题。

GPU 加速解决的是"首次计算太慢"的问题。从性能数据看，首次无缓存时 pose3d 阶段占总耗时的 80% 多，其中的 pose2d 子阶段——也就是 MediaPipe CNN 推理——又占了 pose3d 阶段的 50% 多。如果换成 GPU 推理，CNN 部分确实可以加速 2 到 3 倍。但 MediaPipe 的 GPU 后端在 Windows 上配置很麻烦，而且目标部署机器通常没有 NVIDIA 独立显卡。

缓存解决的是"重复计算太浪费"的问题。在实际使用中，同一个视频被分析多次的概率非常高——换分析档位、修完 bug 重跑、对比不同参数设置。缓存让重复分析的耗时从 200 多秒降到 1 秒左右，这个收益率——约 200 倍——远超 GPU 加速的 2 到 3 倍。而且缓存不依赖任何硬件条件，任何机器上都能受益。

两者不是互斥的，完全可以在 GPU 上跑首次分析，用缓存加速后续分析。我们现在做了后者，如果以后目标部署机器普遍配备了 GPU，再补充前者。

### Q24. FFT 互相关具体是怎么计算两个视频之间的时间偏移的？

**完整答案：**

这个计算分四步。

第一步是音频提取。用 ffmpeg 从两个视频里分别提取音频流，转成单声道 16 位 8kHz 采样的 PCM WAV 文件。8kHz 采样率对语音和蜂鸣器声完全够用，而且数据量小，做 FFT 很快。

第二步是 FFT 互相关。用 `scipy.signal.correlate` 对两段音频信号做互相关，method 参数设为 fft。互相关函数 C(τ) 衡量的是信号 a 在延迟 τ 下与信号 b 的相似度。在时间域算互相关的复杂度是 O(n²)，但通过 FFT 可以降到 O(n log n)。

第三步是找偏移量。互相关序列中最大值的索引记为 peak_idx。因为两段音频长度相等，偏移量是 peak_idx 减去信号长度再加一。这个值可正可负。如果为正，说明相机 A 的音频滞后于相机 B，也就是相机 A 在时间轴上比较晚，我们需要剪掉相机 A 的开头或者说剪掉相机 B 的末尾。具体方向由 `sync.py` 中的裁剪逻辑处理。

第四步是转换为秒。偏移的采样点数除以 8kHz 采样率，就得到了秒级的时间偏移。比如计算出来偏移是 1200 个采样点，那就是 1200 / 8000 = 0.15 秒。

蜂鸣器检测是串行可选的优化步骤。如果音频中存在持续时间超过 2.5 秒、振幅超过最大振幅 40%、且该区间能量占总能量超过 70% 的区间，就判定为蜂鸣器段。我们取蜂鸣器结束的那个采样点作为对齐基准点，而不是用互相关峰值的原始位置。因为蜂鸣器的结束是一个干净、瞬态、清晰的时间标记，用它做对齐比用模糊的互相关峰值更加精确。

### Q25. 三个分析档位——快速、均衡、质量——具体有什么区别？

**完整答案：**

三个档位通过调整两个核心参数来控制分析速度和质量之间的权衡。

第一个参数是 frame_stride，也就是跳帧步长。快速档位 stride 等于 2，每两帧取一帧处理，100fps 的视频等效于 50fps 的时间分辨率。均衡档位 stride 在 1 到 2 之间，部分跳帧。质量档位 stride 等于 1，不跳帧，每一帧都处理。这个参数的设置逻辑是——高帧率视频相邻帧之间差异非常小，隔帧采样不会损失太多信息，但计算量直接减半。

第二个参数是 max_frames，也就是最大处理帧数。快速档位会对长视频做更激进的截断，质量档位会处理更多的帧。这个参数直接控制分析总时长。

另外还有 temporal_filter_enabled 这个开关。时域滤波——就是 median_then_mean 的级联滤波——能有效平滑 3D 轨迹的抖动，但也会消耗额外的计算时间。快速档位可能关闭它来省时，质量档位默认开启。

从实际性能数据来看，同一段 21.6 秒的 100fps 视频在快速档位下总耗时约 40 秒（其中首次无缓存时 pose3d 约 120 秒，有缓存时约 1 秒），总体 realtime_ratio 约 0.5，也就是分析速度是实际视频速度的一半左右。质量档位（stride=1）处理同样的视频，耗时会增加，但时间分辨率翻倍。用户根据自己的需求选择合适的档位——日常训练监控用快速，技术诊断和研究用质量。

## 答辩策略总结

1. **不确定的问题诚实作答**：说"这个我们还没有深入测试，根据我们的理解应该是..."比瞎编强十倍。评委看重的不是你什么都知道，而是你知道自己不知道什么。

2. **主动暴露局限而不是等人问**：在介绍每个模块时顺便提一句局限和改进方向，评委反而会觉得你有批判性思维。

3. **用数据说话**：perf_metrics.json 里有真实的耗时数据——T_sync 几秒、T_pose2d 一百多秒、T_kinematics 三十多秒、realtime_ratio 在 0.09 到 0.57 之间。引用具体数字比"很快"、"挺准的"有说服力一百倍。

4. **核心叙事定调**："这是一个在精度、成本、部署复杂度三者之间取得平衡的方案。它没有用最前沿的算法，但在目标场景——运动实验室的 Windows 台式机——下是可用的、可靠的、可维护的。"
