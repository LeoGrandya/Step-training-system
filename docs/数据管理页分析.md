# 数据管理页面架构分析

> 更新日期：2026-06-08

## 一、页面概览

数据管理页面是一个**按资源类型分栏的通用 CRUD 管理台**，左侧 10 个资源标签页切换，右侧展示对应资源的列表、搜索、筛选、编辑表单。

**新增记录仅在 `creatable !== false` 时显示表单**，编辑记录仅在 `editable !== false` 时工作。不可创建的资源（如 accounts）默认不展示表单。

路径：`/#/admin`（DataManagementView）

## 三、10 个资源清单

| # | key | 标题 | 端点 | 创建 | 编辑 | 删除 | 数据隔离 |
|---|-----|------|------|------|------|------|----------|
| 1 | subjects | 受试者 | /api/v1/subjects | ✅ | ✅ | ✅ | ✅ X-Account-Id |
| 2 | footwork-types | 基础步伐 | /api/v1/footwork-types | ✅ | ✅ | ❌（前端禁用） | ❌ 全局共享，删除按钮已移除以防误删 |
| 3 | routes | 自定义跑动序列 | /api/v1/routes | ✅ | ✅ | ✅（仅自己的） | ✅ X-Account-Id（直接） |
| 4 | training-configs | 训练配置 | /api/v1/training-configs | ✅ | ✅ | ✅ | ✅ X-Account-Id (via subject) |
| 5 | training-videos | 训练视频 | /api/v1/training-videos | ❌ | ✅ | ✅ | ✅ X-Account-Id (via subject) |
| 6 | kinematics-datasets | 运动学数据 | /api/v1/kinematics-datasets | ❌ | ❌ | ❌ | ✅ X-Account-Id (via subject) |
| 7 | evaluations | 效果评估 | /api/v1/evaluations | ✅ | ✅ | ✅ | ✅ X-Account-Id (via subject) |
| 8 | accounts | 账号 | /api/v1/accounts | ❌ | ✅ | ✅ | ❌ 管理员可见全部 |
| 9 | roles | 角色 | /api/v1/roles | ✅ | ✅ | ✅ | ❌ 全局共享 |
| 10 | permissions | 权限 | /api/v1/permissions | ✅ | ✅ | ✅ | ❌ 全局共享 |

### 数据隔离说明

- **subjects**：直接通过 `Subject.created_by_account_id` 过滤
- **training-configs / training-videos / kinematics-datasets / evaluations**：通过 `subject_id → Subject.created_by_account_id` 间接过滤。列表用 `subject_ids` 参数做 DB 级过滤，单条 GET/PUT/DELETE 用 `check_subject_ownership` 校验
- **accounts / roles / permissions**：无账号隔离，全局可见（面向管理员）
- **routes**：直接通过 `RouteDefinition.created_by_account_id` 过滤。LIST 显示当前账号创建的 + NULL 归属（向后兼容旧数据）。PUT/DELETE 校验归属（非本人返回 403）。POST 自动注入 `createdByAccountId`。**注意**：旧端点 `/api/custom-footworks` 暂未接隔离，数据管理页走 `/api/v1/routes` 不受影响
- **footwork-types**：全局共享的步伐字典。前端已禁用删除按钮（`deletable: false`），防止误删核心步伐。后端 DELETE 端点保留供运维调用

## 四、数据流（页面生命周期）

### 4.1 页面挂载（onMounted）

```
1. clearForm()          — 清空编辑表单
2. resetFilters()       — 清空筛选条件
3. loadLookups()        — 遍历所有资源的 lookup 字段，调用 listAdminRecords 预加载下拉选项
   例：subjects → GET /api/v1/subjects?limit=200
       footwork-types → GET /api/v1/footwork-types?limit=200
       routes → GET /api/v1/routes?limit=200
       training-configs → GET /api/v1/training-configs?limit=200
       kinematics-datasets → GET /api/v1/kinematics-datasets?limit=200
4. loadResources()      — 加载当前选中资源的数据列表
   例：subjects → GET /api/v1/subjects?limit=20&offset=0
```

### 4.2 用户操作

| 操作 | 触发函数 | 请求 |
|------|----------|------|
| 切换资源标签 | switchResource(key) | GET {endpoint}?limit=20&offset=0 |
| 搜索/筛选 | resetAndLoad() | GET {endpoint}?keyword=...&limit=20 |
| 翻页 | prevPage/nextPage | GET {endpoint}?limit=20&offset=N |
| 新增（仅 creatable≠false） | saveResource() | POST {endpoint} |
| 编辑（仅 editable≠false） | saveResource() | PUT {endpoint}/{id} |
| 删除（仅 deletable≠false） | deleteResource() | DELETE {endpoint}/{id} |

### 4.3 表单显示条件

```
form v-if="fields.length && (editingId ? editable !== false : creatable !== false)"
```

- 未进入编辑态时，资源 `creatable === false` → 表单隐藏（如 accounts、training-videos）
- 编辑态时，资源 `editable === false` → 表单隐藏（如 kinematics-datasets）
- 点"编辑"按钮会设置 `editingId` 并加载数据到表单

## 五、已解决的问题（2026-06-08 本轮更新）

### ✅ 5.1 数据隔离（training-configs / training-videos / kinematics-datasets / evaluations）

**方案**：
- Repository 层：四个 list 函数新增 `subject_ids` 参数，DB 级过滤 `IN (subject_ids)`
- API 层：每个模块新增 `_account_id()` / `_account_subject_ids()` / `_check_ownership()` 辅助函数
  - LIST：传入 `subject_ids` 做过滤
  - GET/PUT/DELETE：通过 `subject_id` → `check_subject_ownership` 校验
  - POST：创建时校验 `subject_id` 是否属于当前账号
- 无 X-Account-Id 时保持向后兼容（不过滤）

### ✅ 5.2 密码哈希暴露

**方案**：
- 前端：adminData.js 删除 accounts 的 `passwordHash` 字段
- 后端：`serialize_account` 不返回 `password_hash`（原本就如此）
- 后端：`_account_values_from_payload` 新建账号时走 PBKDF2 哈希，编辑时不改密码

### ✅ 5.3 受试者下拉框同名区分

**方案**：
- `normalizeItemLabel()` 优先级调整为 `displayName > name > ...`
- 受试者 `displayName` 已在上一轮通过 `_compute_display_names()` 生成

### ✅ 5.4 RBAC 守卫

**决定**：保持禁用。当前无多角色需求，数据隔离靠业务代码中的 `X-Account-Id` 检查。

### ✅ 5.5 前后端能力对齐

**结论**：经验证，所有 10 个资源的后端均支持 `keyword` 参数。无需隐藏搜索框。在 adminData.js 顶部添加了配置文档注释。

### ✅ 5.6 accounts 与 auth 职责分离

**方案**：
- auth 模块：管注册/登录/改密（PBKDF2 哈希）
- rbac 模块：管账号状态管理/删除（不碰密码）
- adminData.js：accounts 资源 `creatable: false`，创建账号走注册流程
- 表单条件优化：`creatable === false` 时表单隐藏

### ✅ 5.7 footwork-types 前端删除保护 + routes 账号隔离（2026-06-09）

**方案**：
- **footwork-types**：`adminData.js` 设 `deletable: false`，前端隐藏删除按钮。后端 API 不变，DELETE 端点保留供运维调用
- **routes 账号隔离**：
  - `routes.py`：新增 `_check_route_ownership()` 辅助函数。LIST 传入 `X-Account-Id`，POST 自动注入 `createdByAccountId`，PUT/DELETE 校验归属
  - `repositories.py`：`list_routes_page` 新增 `account_id` 参数，过滤 `created_by_account_id = ? OR created_by_account_id IS NULL`（NULL 向后兼容旧数据）；`_route_values_from_payload` 提取 `createdByAccountId`
- 无 `X-Account-Id` 时保持向后兼容（不过滤）

## 六、遗留问题

| # | 问题 | 优先级 | 说明 |
|---|------|--------|------|
| 1 | footwork-types 后端 DELETE 仍开放 | 低 | 前端已禁用删除按钮，但后端 API 未做权限控制。运维可直接调 DELETE 删除基础步伐 |
| 2 | roles / permissions 无账号隔离 | 低 | 全局 RBAC 系统，管理页面仅管理员可用 |
| 3 | adminData.js 未关联到实际 RBAC 权限 | 低 | 页面访问控制未落地，当前通过前端路由入门 |
| 4 | kinematics-datasets 不可编辑/删除 | 低 | 由分析 pipeline 自动生成，手动编辑无意义 |
| 5 | `/api/custom-footworks` 旧端点未接账号隔离 | 低 | 数据管理页不走此路径（走 `/api/v1/routes`），旧端点计划后续统一 |
