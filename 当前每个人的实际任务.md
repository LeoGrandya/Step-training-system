# 当前每个人的实际任务

> 当前架构基线：单 Flask App + MySQL + SQLAlchemy/Alembic，不拆 5 个 Flask 服务，不做 API Gateway。旧 SQLite 数据不保留。  
> 共同规则：所有接口继续返回 `ok/error/items/total/limit/offset` 这一类兼容结构；视频、CSV、JSON 产物继续放文件系统，MySQL 只保存路径、状态、摘要和关键可查询字段。

## 开发顺序

1. 主脑先完成并审核 MySQL schema、Alembic migration、统一分页、统一响应、删除策略和模块边界。
2. 陈彦竹先做受试者和基础步伐，因为它们是后面所有模块的下拉框和外键来源。
3. 雷润华做训练路线和显示逻辑，依赖基础步伐。
4. 金彦廷做训练配置、视频元数据和最小权限，依赖受试者和路线。
5. 许婉其做分析任务和运动学数据索引，依赖视频元数据和现有 pipeline。
6. 郝雨萱做训练效果评估和报告，依赖运动学数据集。

## 陈彦竹：乒乓球步伐及受试者信息管理系统

**目标**：完成受试者信息和乒乓球基本步伐的基础管理，给其他模块提供稳定下拉数据。

**负责表**

- `subjects`：受试者/运动员信息，兼容旧 `/api/users`。
- `footwork_types`：基础步伐字典，如跨步、并步、撤步、两点、三点、四点。

**负责接口**

- 兼容接口：`GET/POST /api/users`、`GET/PUT/DELETE /api/users/<id>`。
- 新增接口建议：`/api/v1/subjects`、`/api/v1/footwork-types`。
- 查询参数统一使用：`keyword`、`limit`、`offset`、`is_active`。

**前端页面**

- 受试者管理页。
- 基础步伐管理页。
- 所有训练、分析、报告页面里的受试者和步伐下拉框。

**不能改动的边界**

- 不要把 `subjects` 当系统登录账号；账号权限属于金彦廷的 `accounts/roles/permissions`。
- 不要物理删除已经被训练、视频、分析任务引用的受试者或步伐；默认使用 `is_active/deleted_at` 软删除。
- 不要私自修改 `/api/analysis/jobs` 或报告接口。

**验收标准**

- 受试者支持增删改查、数据校验、多条件模糊搜索、分页显示。
- 基础步伐支持增删改查、下拉选择、数据校验、模糊搜索、分页显示。
- 删除被引用数据时不会导致历史分析和报告断链。
- 验收命令：`cd frontend; npm.cmd test; npm.cmd run build`。

## 雷润华：乒乓球步伐训练指导及显示系统

**目标**：完成训练路线定义、动作要求展示、节奏控制和实时显示需要的数据结构。

**负责表**

- `route_definitions`：具体训练路线方案，承接当前自定义步伐。
- `route_steps`：路线中的每一步，记录宫格、顺序、动作类型和节奏。

**负责接口**

- 兼容接口：`GET/POST/PUT/DELETE /api/custom-footworks`。
- 新增接口建议：`/api/v1/routes`、`/api/v1/routes/<id>/steps`。
- 查询参数统一使用：`keyword`、`footwork_type_id`、`limit`、`offset`。

**前端页面**

- 训练路线定义页。
- 训练路线显示页。
- 节奏调控控件。
- 当前训练页中的自定义路线选择区域。

**不能改动的边界**

- 不要新建另一套 `custom_footworks` 主表；旧接口只作为 `route_definitions` 的兼容入口。
- 不要直接操作视频上传、运动学数据或评估记录。
- `route_steps` 可随路线级联删除，但已被训练记录引用的路线本身不能物理删除。

**验收标准**

- 可自定义跑动路线、开始宫格、每步顺序和节奏。
- 路线支持增删改查、下拉选择、数据校验、多条件模糊搜索、分页显示。
- 训练显示能读取路线步骤并稳定渲染。
- 验收命令：`cd frontend; npm.cmd test; npm.cmd run build`。

## 金彦廷：乒乓球步伐训练数据管理系统

**目标**：完成训练参数配置、训练视频元数据管理和最小权限体系。

**负责表**

- `training_configs`：训练参数配置。
- `training_videos`：左右机位视频元数据，只保存路径和关联关系。
- `accounts`、`roles`、`permissions`、`account_roles`、`role_permissions`：最小 RBAC 权限。

**负责接口**

- 新增接口建议：`/api/v1/training-configs`、`/api/v1/training-videos`、`/api/v1/accounts`、`/api/v1/roles`。
- 兼容分析上传：`POST /api/analysis/jobs` 上传后必须能写入 `training_videos`。
- 查询参数统一使用：`subject_id`、`route_definition_id`、`status`、`limit`、`offset`。

**前端页面**

- 训练参数配置页。
- 训练视频管理页。
- 权限管理页。
- 上传视频页面中的训练配置选择区域。

**不能改动的边界**

- 视频文件不进 MySQL；只保存 `left_video_path`、`right_video_path`、原始文件名、大小、状态。
- 权限只做最小可用：管理员、成员、只读；不要引入复杂组织架构。
- 前端隐藏按钮不等于权限，后端接口必须校验。
- 不要改动 pose3d、video_sync 或 kinematics 算法。

**验收标准**

- 训练参数支持配置、修改、查询、统计和分页。
- 视频上传后 `training_videos` 有左右视频路径和受试者关联。
- 权限在后端接口生效。
- 验收命令：`cd frontend; npm.cmd test; npm.cmd run build`。

## 许婉其：乒乓球步伐训练关节运动学数据管理系统

**目标**：把现有分析任务和运动学产物接入 MySQL，形成可查询的数据索引。

**负责表**

- `analysis_jobs`：分析任务状态主表。
- `kinematics_datasets`：运动学 CSV/JSON 产物索引和摘要。
- `kinematics_frame_metrics`：少量关键逐帧指标或抽样数据。

**负责接口**

- 兼容接口：`POST /api/analysis/jobs`、`GET /api/analysis/jobs/<job_id>`、`GET /api/analysis/jobs/<job_id>/result`。
- 新增接口建议：`/api/v1/kinematics-datasets`、`/api/v1/kinematics-datasets/<id>/metrics`。
- 查询参数统一使用：`subject_id`、`status`、`stage`、`start_date`、`end_date`、`limit`、`offset`。

**前端页面**

- 分析任务状态页。
- 运动学数据管理页。
- 关节运动学数据详情页。

**不能改动的边界**

- 不重写现有分析算法；只接入状态和产物索引。
- 完整 CSV 不全量塞进 MySQL；完整数据仍保留在文件系统。
- `JobStore/meta.json` 暂时保留为运行时调试和恢复辅助，MySQL `analysis_jobs` 是业务主索引。
- 不要创建和郝雨萱重复的评估结果表。

**验收标准**

- 上传分析后 `analysis_jobs` 有任务状态、进度、阶段、错误信息。
- pipeline 完成后 `kinematics_datasets` 有 `frame_metrics.csv`、`session_summary.csv`、`step_metrics.csv`、`unit_metrics.csv` 路径。
- 关键运动学指标可查询、可分页、可关联受试者和视频。
- 验收命令：`cd frontend; npm.cmd test; npm.cmd run build`。

**负责表**

**前端页面**

- 训练效果评估页。
- 历史报告页。

- 评估必须基于 `kinematics_datasets` 和现有报告 payload，不直接重新跑视频分析。

**验收标准**

## 全组共同验收

- Alembic 可以从空 MySQL 库执行首版建表。
- 所有旧接口 URL 不变，旧前端不会因为返回结构变化而断裂。
- 被历史任务引用的数据不允许随便物理删除。
- JSON 只用于快照和摘要；搜索、统计、分页字段必须建普通列和索引。
- 最终全链路必须跑通：受试者 -> 步伐/路线 -> 训练配置 -> 视频上传 -> 分析任务 -> 运动学数据 -> 效果评估 -> 历史报告。
## Schema 修复补充（2026-06-04）

- 全组统一按 MySQL 5.5 兼容 schema 开发：JSON 字段在 ORM 使用 `JsonText`，数据库实际为 `TEXT`。
- 雷润华负责的 `route_definitions` 活跃唯一性使用 `active_name_sequence_hash VARCHAR(64) NULL UNIQUE`，不是 `name_norm + sequence_canon` 复合唯一索引。
- 路线创建/更新时写入 SHA-256 hash；路线软删除时清空 hash，允许重新创建同名同路线；重复活跃路线仍必须返回 409 或触发唯一冲突后转为 409。
